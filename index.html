Project: ATS1665 - Understanding Gender 1

Make the highest quality online undergraduate unit possible

This unit was a first of its kind in the Faculty of Arts. The academic Frankie Egan, coordinating a unit for the first time, and I were given the challenge to design the ideal online learning unit that will set the example for a new generation of online offerings as a directive from the Associate Dean of Arts. This was in response to the lockdowns at the start of the pandemic where many students were forced to miss out on units that required face-to-face teaching; this unit would be one of the many alternatives offered in Arts.

  • Faculty of Arts senior academics
  • Frankie Egan - Unit Coordinator
  • Educational Design Team
  • Teaching Support Team
  • Students enrolling in the unit

My responsibilities were:

  • To collaborate with Frankie on an assessment schedule, the teaching and learning content and a communication strategy for the unit
  • Train Frankie on any educational technology required for the project
  • Report time taken at various stages of the project
  • Review the project once complete and build out a template and guides for how we achieved what we did
  • Monitor weekly progress and update content in the unit site as requested by Frankie

Assessment:

  • 4 quizzes (1 every 3 weeks) - take top 2 results - 20%
  • Completion of weekly Moodle lessons - H5P tasks + activity completion log - 10%
  • 6 blog posts (250 words each) on Mahara - 30%
  • 1 essay (1500-2000 words) based on chosen blog post - 40%

Learning content:

  • 10 weekly lessons + 2 live guest lectures/Q+A (extended to 4 guests)
  • Highly visual content with videos of lectures, TED Talks, and interviews
  • Weekly content had basic, intermediate, and advanced levels to choose from

Results:

  • Highly interactive and bespoke approach
  • Highest rating for a first-time unit coordinator and highest rating for an Arts Faculty unit delivered solely online
  • Positive response from students with some suggestions for improvement

Project: BEEST

Simplifying advanced HTML element design

BEEST stands for Bootstrap Enhanced Elements Streamlining Tool. BEEST takes your basic input of text, links and choices and combines it with pre-written HTML to generate advanced HTML elements for Moodle without the need for complex HTML skills. Take a look at the current state of the project here.

  • Academics building course content
  • Senior academics in the Faculty of Arts seeking ways to improve learning
  • The Education Design teams in both Arts and IT
  • Students
  • Teaching Support in the Faculty of Arts

I designed the very first element creator - the button builder - with my limited knowledge of HTML, CSS and Javascript. I used this basic example to show two Educational Designers in the IT Faculty - Matt Chen and Philip Abramson - in IT with exceptional Educational Design, User Experience and coding skills - Matt a former front end web designer, Phil a former back end CMS expert - and convince them it was a worthwhile prospect building all the other elements you see on the BEEST.

From here Matt and Phil built out the functionality, storage and hosting of the site, and I worked on the following:

  • Researching and implementing the user interface (UI) of the BEEST. I built and iterated the interface several times as I gained user feedback and research how people like to use applications like this.
  • I came up with the idea for naming and storing items for reuse, and Phil made this happen while also teaching me how to access local storage in the browser.
  • I worked with Matt to streamline the functionality so the BEEST works quicker with the user, learning about extraction of functionality and more advanced Javascript principles.
  • I recorded all the walkthrough screencasts and wrote all the user guides.
  • I coordinated 4 focus groups - other Educational Designers, Academics from Arts, Students and Senior Teaching and Learning staff - to gain and incorporate their feedback into the product. In the process winning many senior staff over that we can trust staff with working with code, that the product is self-sufficient.
  • Setup the Google Analytics to track use of the product. This was instrumental in proving self sufficiency. We audited 50 sites that used the BEEST in it's first semester of being public, the site was used in excess of 1000 times with only 4 inquiries for support, and only 6 instances where we found content from the BEEST that was used inappropriately - e.g. an accordion inside of another accordion.

Over 500 academics and professional staff across the university still use BEEST. But the real win here is that the central teaching and learning unit saw how useful BEEST was, and how many people were using it, and using it well without requiring the levels of support they expected. They decided to create their own tool, based off the BEEST that was built in to Moodle's backend rather than continuing to put BEEST in on the front end.

The BEEST solved many problems in learning design to improve learning and user experience:

  • The big one in our eyes. All content built with the BEEST is responsive across devices. Students can now use Moodle with confidence on laptop/desktop, tablets and mobile.
  • It cuts out the need for staff to learn complex HTML to build complex HTML elements.
  • It harnesses in built Bootstrap and Font Awesome library in Moodle to create these elements. It only adds extra CSS for Accordions and Tabbed content to ensure it works correctly in Moodle. All other elements require no additional external CSS or Javascript to operate.
  • Elements can be saved and revisited and updated. Making elements reusable is another user efficiency we were able to make happen using the browser's local storage capacity. Each item is around 1-4kb of memory and most browsers allow 10MB of local storage per site. Meaning we can store between 2500-10000 BEEST items per user in their browser and not pay for server space.
  • It allows us as Educational Designers to speak more to a user experience perspective of learning. Asking questions like 'is how you have formatted this really allowing to student to get the best out of their time?' This is what learning online is missing in many cases where it goes wrong. A user experience that is clear, precise and smooth for the learner. BEEST can help here to reduce the learning curve for newcomers.

Project: MD Design - Postgraduate Studies in Medicine

Transforming how future doctors study

UQ's Medicine Faculty is redesigning its postgraduate medicine course to be 4 x 1-year-long courses, rather than 32 individual courses. This means we are combining the themes, content and assessment of 8 x 12-week courses into 1 x 32-week course. This redesign has been prompted by doctors in the field and current students raising concerns that the areas of learning are not integrated well enough. Students are out practicing using compartmentalised knowledge, rather than applying their skills holistically to each given situation. In an attempt to rectify this, the entire program has been reduced to its fundamentals for year one and put back together again.

The stakeholders in the Faculty of Medicine are:

  • Academics collaborating on assessment and learning content and delivering classes
  • Professional staff developing and maintaining learning content
  • Administrative staff responding to student and staff inquiries
  • The MD Project Team leading the design and development of the new program

The stakeholders in the broader context of the university are:

  • Learning designers and academics in other schools outside of the Faculty of Medicine; the School of Biomedical Sciences, the School of Public Health, and the School of Philosophy.
  • The central teams supporting the project team, academics, and professional staff; Digital Learning Uplift team, eLearning Support, IT Support
  • The students studying the new program

This is my current full-time learning design role. I am currently working on the assessment regime for year two, and developing content in collaboration with the stakeholders above for year one and two in several subsections of the program:

  • The Assessment Working Group; designing and developing the assessment schedule for years 1-4
  • Clinical and Practical Skills
  • Advocacy and Public Health
  • Ethics and Law
  • Rural and Remote Medicine

I am also using the learnings from my project on the BEEST at Monash to bring responsive design and incorporating quality user experience into the learning design to UQ's Medicine course. I have designed a similar application for UQ's LearnX and added Bootstrap, FontAwesome and custom CSS and JS to the platform to produce a LearnX course that is responsive across devices. The working title is the MEDI7100 Code Centre - view the code centre here. I have built, iterated and continue to maintain this web application. Because it is built based on Bootstrap v5.2, and we are referencing this library in LearnX, there will be very little maintainence required due to the static nature of this library. I have begun to develop Professional Development (PD) for academic and professional staff, which includes recent research around user experience so that they can utilise this web application to build learning that meets their brief, plus is an interesting user experience for the students without the need to code.

Initially, students and staff were dubious of the program's success. Integrating such a broad spectrum of knowledge and skills while maintaining the standards set by UQ would result in chaos, and frustrated stakeholders. After 18 months of planning, and 10 weeks of the program being delivered, opinions of all the critical stakeholders has shifted to accept the change, and students are excited. With the advent of a project team we are also getting a much quicker feedback loop with students on content organisation and assessment. This means closing feedback loops in hours, instead of weeks or whole iterations of a course.

Students see the results of providing feedback and continue to be open with us about improving their experience. To me, this close relationship with the student base is the key to success of the change. If academics see students engaging and enjoying themselves, regardless of how much work is required to continue the program in this fashion they will do it. While there are no definitive results of success 10 weeks into the delivery, the murmurs of unwinding the changes and returning to the old delivery from academics and student reprasentatives alike have stopped.

Project: Music Performance in the Pandemic

Automating Google Drive for the good of music

When all students and staff were sent home from Monash, music and performance students and staff faced a unique issue. So many of the courses relied on assessment of a live performance component. With the inability to be in the same space how would musicians form and perform in a band for their assessment? Even if they could, how would they submit? How would that submission be disseminated to relative markers? Enter myself as an Educational Designer in Arts working with the Music Faculty and Matt Chen, an Educational Designer from IT. Between us we used Google Suite and Apps Scripts to automate a huge amount of online work for music and drama assessment during Semester 1 of 2021, which prevented a complete shut down of the courses at the time.

  • Music and theatre students
  • Music and theatre staff
  • Arts faculty senior academics and administrative staff
  • Arts teaching support professional staff

I wireframed the entire onine assessment and marking process for music performance, which was then applied to the theatre courses as well. The steps involved:

  1. Sell my idea for online performance assessment to the Associate Dean for Education in Arts, Dean of the Music School and leader of Teaching Support for the Faculty of Arts.
  2. Engage Matt for support to execute the scripts required to fire off several steps automatically across the Google Suite
  3. Gather student data for assessment:
    • Student repertoire for the 6 performance units/200 students across 6 semesters
    • Connect students to relative markers. In performance at least 2 markers for each performance are required. A Monash staff member plus an instrumental specialist for the instrument the assessed student is playing

From here, the step-by-step process an assessment would pass through was as follows:

  1. Submit their repertoire listing for their performance unit, including their instrument and any accompanying artists via a Google Form
  2. Google then automatically created a folder for each student with subfolders for each piece of their repertoire plus a for charts
  3. Each student then received an automated email linking to another form to upload their video-recorded performances
  4. Once the folders from step 2 were all populated with videos:
    • Students were sent a view link to their Google Drive folder to submit to Moodle
    • Matt and I were notified of the submission end date for each unit group. On this date, we used data that had all the markers' names, emails, students they were to mark, and instrumental specialty if they had one. We then automated injecting sharing permissions into the folders they were meant to have access to
  5. Once permissions were injected, Google sent markers a document with a list of students to mark and dates that marks were due
  6. Once the date for markers to be completed was passed, students were then sent an automated email with a link to their marks and feedback
  7. Note: Matt, the course coordinator, and I all had the ability to manually override due dates for individual students for extensions and markers to extend the individual marking deadline

First and foremost, courses continued, students didn't get delayed, casual and contract staff didn't lose their jobs and all stakeholders gained incredible new skills to play and record together online. Secondly, the Faculty gained a highly automated process in the future for when students weren't able to perform live for whatever the reason may be. And finally, Matt, the music and theatre staff, and I all gained an incredible amount of experience dealing with high-stress students and technology none of us had used to this extent before.

While overall compared to years where students could attend face to face, the quantitative feedback was down from an average of half a point, the comments from students in the qualitative feedback was overall incredibly grateful to all the staff for getting the job done in an incredibly difficult time, and not delaying students' progress further.

Staff were overwhelmed with what Matt and I were able to achieve using the Google Suite. That two people could automate the work that should have had a team dedicated to it was something of a miracle, the right people with the right attitude and skillset in the right place at the right time. We minimised revenue loss, job losses, and time lost and created a process that was essentially reduced to clicking a few buttons at the right time to produce automations for the cohort. A win I am immensely proud of being a music student and musician myself.

Research: Building a Quality Balance in Teaching and Research

A systematic approach to improving teaching and learning in higher education

Synopsis

This paper explores the history of higher education (HE) and what has led to the current tense state, where instead of a balanced approach to managing and implementing quality teaching and research, there is pressure either directly or indirectly to favour research by HE providers. My approach was not to simply say that higher education needs to implement teaching heavy positions. Instead, it was to dive deeper into what makes research worthy of such favour, and how higher education in Australia can implement a similar path to balance value for teaching. The main discovery was systems that are in place to measure research, and rank universities based on their research are a priority of the universities themselves. If a rigorous system of reviewing and ranking teaching and learning quality could be created and put in place, we would see external ranking systems by independent bodies rank universities on teaching and learning quality as well as research quality. The main limitations of this study were the unclear nature of financial reports in relation to funding and expenditure on teaching and learning, and an absence of literature on higher education institutions implementing processes for measuring teaching and learning quality. Research: Experience and Reflection: Learning in Action

A systematic approach to improving teaching and learning in higher education

Synopsis

Exploring Experiential Learning (EL) and Community Service Learning (CSL) further to improve my own practice as an educator was the focus of this paper. While experiential learning is becoming more prevalent in its many forms such as Problem/Project Based Learning (PBL) and Social Justice Learning (SJL) I wanted to further understand how the theories that underpin these practices could possibly be applied to the international student experience in Australia. My findings were as follows. To implement a mentoring program and language exchange in the institution I was employed at would help solve two major issues; language acquisition and development, and exploitation of students in unfamiliar work and study conditions. All too familiar are stories of international students working 50+ hours a week, cash in hand, while studying full time, 20+ hours per week, to support themselves.

Mentors who have been in the country for 2+ years could provide new students incredible insight into navigating Australia's system to ensure fairness and safety while in the country. Secondly, language acquisition, while encouraged in the student's home country to achieve a high enough level of confidence with language to attain a position in an Australian institution, tends to come to a halt once enrolled in study due to many competing factors for a student's time and also cultural differences. A language exchange would see local and international students share language knowledge, build relationships and counter the stagnation of language acquisition while also familiarising participants with unfamiliar cultural expectations and norms. What did I use to build this site?

Github

Github is an online development hub. It also has a public pages space so you can build web pages - like this one - and applications - like BEEST and the Code Centre - all for free. The only catch with the free version is everyone gets to see your code, but that's what github is all about. Collaborating and innovating. Taking what other people have built and making it work to solve a problem.

Bootstrap

Bootstrap is the most referenced design library on the internet. I have used it to style and format this site, the BEEST, the Code Centre and many other projects to improve user experience, look and feel of online products. Without Bootstrap this would all be a bunch of plain text on a page. Boring!

Font Awesome

Font Awesome is the most highly used icon library on the internet. It was one of the first to be created, and has stood the text of time to get to version 6. Fun fact: it is the reason why so many icon libraries use the <i> element to display icons and why this element is no longer the standard for italics - instead we use the <em> element. The free version gives you access to just over 2000 icons. The full version has just over 20000 icons. It can take a good project and make it great.
